## What Are ABAP File Formats?

ABAP File Formats (AFF) is SAP's open specification for serializing ABAP repository objects to files. It defines how each object type is split into **JSON metadata** + **source code files**, enabling file-based ABAP development.

**Design principles:**
- **Complete** — file content is sufficient to share/distribute ABAP objects
- **Human-readable** — mirrors what developers see in ADT/Eclipse
- **Clean** — no dates, usernames, system-specific info, redundant/autogenerated content

**Who uses AFF:**
- **ADT for VS Code** (upcoming Q2 2026) — primary file format for the virtual workspace
- **abapGit** — progressively adopting AFF for newer object types (replacing legacy XML format, requires SAP_BASIS >= 7.56)
- **gCTS** (Git-enabled Change and Transport System) — SAP's native Git integration
- **abaplint** — understands AFF file structures

## How AFF Serialization Works

Every ABAP object produces **at minimum one JSON metadata file**: `<name>.<type>.json`. Objects with source code produce **additional source files** (`.abap`, `.acds`, `.abdl`).

### Example: Class (CLAS) → up to 7 files

```
zcl_my_class.clas.json              ← metadata (description, language, category, method/attribute descriptions)
zcl_my_class.clas.abap              ← main ABAP source (public/protected/private sections, method implementations)
zcl_my_class.clas.definitions.abap  ← local type definitions
zcl_my_class.clas.implementations.abap ← local type implementations
zcl_my_class.clas.macros.abap       ← macros
zcl_my_class.clas.testclasses.abap  ← unit test classes
zcl_my_class.clas.texts.en.properties ← text elements (per language)
```

### Example: CDS View (DDLS) → 2 files

```
zmy_cds_view.ddls.json   ← metadata (description, language, sourceOrigin)
zmy_cds_view.ddls.acds   ← CDS source code
```

### Example: Behavior Definition (BDEF) → 2 files

```
zmy_bdef.bdef.json   ← metadata
zmy_bdef.bdef.abdl   ← BDEF source
```

### Metadata-only objects (e.g., SRVB, DOMA, CHKV) → 1 file

```
zmy_service.srvb.json   ← all properties in JSON
```

### JSON Schema Structure

All schemas follow JSON Schema Draft 2020-12, auto-generated from ABAP interface types:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "formatVersion": "1",
  "header": {
    "description": "My Class Description",
    "originalLanguage": "en",
    "abapLanguageVersion": "cloudDevelopment"
  }
}
```

### Translation Files

Text elements use `.properties` format with JSONPath keys:

```properties
$.header.description=Meine Klasse
$.descriptions.methods[?(@.name=='MY_METHOD')].description=Meine Methode
```

### Naming Conventions

- Pattern: `<object_name>.<object_type>.[<sub_object>.<sub_type>.][<content_type>.][<language>.]<extension>`
- Namespaces: `/NMSPC/CL_OBJECT` → `(nmspc)cl_object`
- All lowercase

## How ARC-1 Currently Handles ABAP Objects

Understanding the current architecture helps identify where AFF adds value.

### Reading Objects (`SAPRead`)

- Returns **raw plaintext source** from ADT REST API (e.g., `GET /sap/bc/adt/oo/classes/{name}/source/main`)
- No metadata extraction (description, language version, class category are not returned)
- No structural decomposition (test classes, local definitions, etc. returned as one blob)

### Writing Objects (`SAPWrite`)

- Sends **raw plaintext source** via `PUT` to `/source/main` as `text/plain`
- No metadata serialization
- No structural packaging

### Context Compression (`SAPContext`)

- Custom text format with comment-based headers:
  ```
  * === Dependency context for ZCLASS_NAME (3 deps resolved) ===
  * --- ZIF_INTERFACE (intf, 5 methods) ---
  [interface contract extracted]
  ```
- 7–30x compression vs full source
- Locked to plaintext — no structured/parseable output

### XML Parsing (`xml-parser.ts`)

- Parses ADT XML responses for search results, package contents, diagnostics
- Read-only and extractive (XML → plain objects)
- No XML serialization for writing

### abapGit

- ARC-1 detects abapGit availability via feature probing (`/sap/bc/adt/abapgit/repos`)
- No abapGit integration tools — detection only

## Where AFF Would Add Value to ARC-1

### 1. Structured Object Metadata in SAPRead Responses

**Current problem:** When an LLM reads a class via `SAPRead(CLAS, ZMY_CLASS)`, it gets raw ABAP source with no metadata. The LLM doesn't know the class description, language version, category, or which methods are public vs private without parsing the source.

**AFF solution:** Return AFF JSON metadata alongside source:

```json
{
  "metadata": {
    "formatVersion": "1",
    "header": {
      "description": "Sales Order Processing",
      "originalLanguage": "en",
      "abapLanguageVersion": "cloudDevelopment"
    },
    "classCategory": "00",
    "fixPointArithmetic": true
  },
  "source": "CLASS zcl_sales_order DEFINITION PUBLIC..."
}
```

**Benefit:** LLM gets structured context without parsing ABAP source. Better code generation, better explanations.

**Implementation:** Add `format` parameter to `SAPRead`: `SAPRead(CLAS, ZMY_CLASS, format=json)`. Parse ADT XML metadata response into AFF-compatible JSON using the published schemas.

### 2. Multi-Object RAP Stack Creation

**Current problem:** Creating a RAP service requires 5–10 sequential `SAPWrite` calls (CDS view → projection → BDEF → service definition → service binding → behavior pool). Each call is independent — no transactional guarantee, no dependency ordering.

**AFF solution:** Accept a structured package describing the full stack:

```json
{
  "objects": [
    { "type": "DDLS", "name": "ZI_SALESORDER", "source": "..." },
    { "type": "DDLS", "name": "ZC_SALESORDER", "source": "..." },
    { "type": "BDEF", "name": "ZI_SALESORDER", "source": "..." },
    { "type": "SRVD", "name": "ZUI_SALESORDER", "source": "..." },
    { "type": "SRVB", "name": "ZUI_SALESORDER_O4", "metadata": { "bindingType": "odata_v4_ui" } }
  ],
  "activationOrder": ["ZI_SALESORDER", "ZC_SALESORDER", "ZI_SALESORDER", "ZUI_SALESORDER", "ZUI_SALESORDER_O4"]
}
```

**Benefit:** Directly enables the `generate-rap-service` skill (J4D's "OData UI Service from Scratch" equivalent). Reduces tool calls from ~10 to 1.

**Implementation:** New `SAPWrite` action `batch` or new intent `SAPBatch`. Orchestrates create → activate in dependency order.

### 3. Structured SAPContext Output

**Current problem:** `SAPContext` returns custom plaintext format. While token-efficient for LLM consumption, it's not parseable by other tools and not reusable across conversations.

**AFF solution:** Add JSON output option:

```json
{
  "target": { "type": "CLAS", "name": "ZCL_SALES_ORDER" },
  "dependencies": [
    {
      "type": "INTF",
      "name": "ZIF_SALES_ORDER",
      "contract": {
        "methods": [
          { "name": "CREATE_ORDER", "visibility": "public", "parameters": [...] }
        ]
      }
    }
  ],
  "stats": { "found": 5, "resolved": 3, "failed": 2, "tokens": 47 }
}
```

**Benefit:** Structured dependency graphs enable: offline analysis, version tracking, IDE integration, cross-conversation reuse.

**Implementation:** Add `outputFormat` parameter to `SAPContext`: `text` (default, current behavior) or `json`.

### 4. Test Class Separation

**Current problem:** When reading a class with `SAPRead`, the test class is either included in the blob or requires a separate read of the test include. The LLM can't easily distinguish production code from test code.

**AFF solution:** Return class source decomposed into AFF's multi-file structure:

```json
{
  "main": "CLASS zcl_sales_order DEFINITION...",
  "testclasses": "CLASS ltcl_sales_order_test DEFINITION...",
  "localDefinitions": "...",
  "localImplementations": "..."
}
```

**Benefit:** LLM can reason about production code and test code separately. Critical for the `generate-abap-unit-test` skill — the LLM needs to see existing tests without them being mixed into the production source.

**Implementation:** Fetch class includes via ADT (`/includes/testclasses`, `/includes/definitions`, etc.) and return as structured response.

### 5. Schema-Driven Object Validation

**Current problem:** When the LLM generates ABAP code via `SAPWrite`, validation only happens server-side after the write (syntax check, activation). Invalid metadata or structural issues cause hard-to-debug errors.

**AFF solution:** Validate object metadata against AFF JSON schemas **before** sending to SAP:

```typescript
import Ajv from 'ajv';
const ajv = new Ajv();
const clasSchema = require('@sap/abap-file-formats/file-formats/clas/clas-v1.json');
const validate = ajv.compile(clasSchema);

if (!validate(metadata)) {
  throw new AdtSafetyError(`Invalid class metadata: ${validate.errors}`);
}
```

**Benefit:** Fail fast with clear error messages instead of cryptic ADT XML errors. Especially valuable for LLM-generated code where metadata errors are common.

**Implementation:** Bundle AFF schemas (they're MIT-licensed) and validate in `SAPWrite(create)` before hitting the ADT API.

### 6. Git/abapGit Interoperability

**Current problem:** ARC-1 has no file-based import/export. Objects can only be read/written one at a time via ADT REST APIs. No integration with Git workflows.

**AFF solution:** Add export/import in AFF format:

```
SAPExport(objects: [{type: "CLAS", name: "ZCL_FOO"}, {type: "INTF", name: "ZIF_FOO"}], format: "abap-files")
→ Returns AFF-formatted file structure that can be committed to Git

SAPImport(source: "<AFF file contents>", format: "abap-files")
→ Deserializes AFF files and creates/updates objects via ADT
```

**Benefit:** Bridge between ARC-1's API-driven model and file-based Git workflows. Compatible with both abapGit and gCTS repositories.

**Implementation:** New `SAPExport` / `SAPImport` tools or new `SAPWrite` / `SAPRead` format options.

### 7. Transport Manifest Serialization

**Current problem:** `SAPTransport` returns transport info as text. No structured manifest of what's in a transport.

**AFF solution:**

```json
{
  "transport": "K900123",
  "description": "Sales Order RAP Service",
  "owner": "DEVELOPER1",
  "status": "modifiable",
  "objects": [
    { "type": "CLAS", "name": "ZCL_SALES_ORDER", "action": "insert" },
    { "type": "DDLS", "name": "ZI_SALESORDER", "action": "insert" }
  ]
}
```

**Benefit:** LLM can reason about transport contents, suggest missing objects, check consistency.

## What NOT to Do

| Anti-pattern | Why |
|-------------|-----|
| **Recreate abapGit's file-to-Git workflow** | abapGit already does this well. ARC-1 should focus on API-level orchestration, not file-based version control. |
| **Full object metadata preservation** | ADT REST APIs don't expose all metadata (revision history, user comments). AFF can't recover what the API doesn't provide. |
| **Replace the current plaintext SAPRead output** | The plaintext format is token-efficient for LLMs. AFF JSON should be an *additional* format option, not a replacement. |
| **Implement AFF serialization from scratch** | Use the published JSON schemas as validation references. For actual serialization, parse ADT XML responses into AFF-compatible JSON — don't try to replicate SAP's ABAP-side AFF framework in TypeScript. |
